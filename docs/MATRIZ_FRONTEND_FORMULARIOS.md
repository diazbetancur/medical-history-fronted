Base: [AUDITORIA_FORMULARIOS_POST_PUT_PATCH.md](/d:/Proyects/AppLand/Proyects/Medical-app/medical-history-backend/AUDITORIA_FORMULARIOS_POST_PUT_PATCH.md)

## Convenciones para frontend

- `Oblig. UI (*)`: solo indica si el label debe llevar `*`.
- No usar placeholders para comunicar obligatoriedad.
- `Contexto`: `Crear`, `Editar`, `Ambos`.
- `Cond.` = condicional.
- Si una columna queda `-`, no se detecto regla explicita en backend.

### 1. Campos obligatorios reales para labels con `*`

- Registro: `email`, `password`, `confirmPassword`, `firstName`, `lastName`

- Antecedente: `type`, `title`
- Alergia: `allergen`, `severity`, `status`

- Excepcion disponibilidad: `type`, `startDateTime`, `endDateTime`
- Sede profesional: `name`

- Admin user roles replace: `roles[]`

- RBAC user lock: `locked`
- Role create: `name` aunque el DTO no lo marque
- Role permissions: `permissions[]` si la UX no trabaja con estado vacio implicito

### 2. Campos condicionales

- `confirmPassword`: obligatorio si hay `password`.
- `confirmNewPassword`: obligatorio si hay `newPassword`.
- `endDate` en medicacion: debe quedar vacio si `isOngoing = true`.
- `overrideStartTime` y `overrideEndTime`: obligatorios si `type = Override`.
- `weeklyWindows[].institutionId` / `weeklyWindows[].professionalLocationId`: idealmente usar uno u otro, no ambos.
- `serviceId` en solicitud publica: opcional, pero si existe debe pertenecer al `profileId`.
- `slug` del perfil profesional: opcional; si no se envia, backend lo genera.
- `officialName` en approve proposal: opcional; si no se envia, backend usa `ProposedName`.
- `durationMinutes` en cita propia: no debe pedirse al usuario; backend lo deriva del template.
- `patientId`, `patientName`, `patientEmail`, `patientPhone` en cita propia: no deben mostrarse ni enviarse desde FE.

### 3. Inconsistencias que frontend debe respetar temporalmente

- Passwords: DTO publico/admin dice minimo 6, pero Identity exige minimo 8 + mayuscula + minuscula + numero + simbolo.
- Perfil profesional:
  - `businessName`: DTO 200, DB 150
  - `slug`: DTO 250, DB 150
  - `address`: DTO 500, DB 300
- Servicios profesionales:
  - `name`: DTO 200, DB 150
- Solicitud publica:
  - `clientEmail`: DTO 150, DB 100
  - `clientName`: DTO 100, DB 150
  - `message`: DTO 1000, DB 2000
- Notas de requests:
  - DTOs admin/profesional limitan 500
  - DB permite 1000
- Perfil de paciente:
  - backend no valida coherencia `countryId` vs `cityId`
  - FE debe resolver ciudad dependiente del pais
- Perfil profesional:
  - backend valida pais y ciudad activos, pero no valida que la ciudad pertenezca al pais
- Disponibilidad:
  - `AvailabilityController` no valida ownership del `professionalId`
  - FE no deberia construir URLs arbitrarias desde cliente
- Excepcion de disponibilidad:
  - `institutionId` existe en DTO, pero hoy no se persiste
  - FE no debe depender de ese valor
- Medicaciones:
  - `status` invalido se convierte silenciosamente a `Active`
  - FE debe forzar select cerrado
  - backend no valida `endDate >= startDate`
- Educacion:
  - backend no valida `graduationYear >= startYear`
  - FE debe validarlo
- Citas update / reschedule:
  - varios strings no tienen semantica clara de "clear"
  - `null` suele significar "mantener actual"
- RBAC lock:
  - `locked=true` solo cambia `LockoutEnabled`
  - no configura `LockoutEnd`
  - FE no debe prometer bloqueo efectivo inmediato
- Favoritos publicos:
